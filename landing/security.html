Security

How we secure Dendra

Specific commitments, dated. We list what we do today, what we don't yet claim, and how to report a vulnerability. Updated 2026-05-13.

What we do today

SDK + build pipeline

  • No pull_request_target triggers anywhere in the repo. The most-exploited workflow trigger in recent supply-chain attacks is not part of our CI surface.
  • OIDC publish is tag-gated. Our PyPI release workflow has id-token: write only on push: tags: ['v*'] and workflow_dispatch. Fork PRs cannot trigger a publish.
  • No actions/cache in workflows that publish. There is no cache-trust-boundary that an attacker could poison across PR and release runs.
  • npm ci, not npm install in CI. The committed lockfile state is enforced; an attacker can't substitute a different transitive version through a resolver trick.
  • Provenance attestations on every PyPI release via the GitHub Actions trusted-publisher binding + Sigstore. Verify with pip install --require-hashes or by checking the PyPI project page for the attestation.
  • Signed tags + commits. Release tags are signed with the maintainer's Ed25519 SSH key; commits on main and on tagged releases are signed. git verify-tag v1.1.0.
  • Dependabot alerts + Push Protection on the repo. HIGH / CRITICAL alerts get same-week dependency bumps. The 2026-05-11 audit closed 16 alerts including the App Router middleware bypass (CVE-2026-45109).

Cloud surface (api.dendra.run + app.dendra.run)

  • TLS in transit, Cloudflare-managed at-rest encryption on D1. Edge-resident; no customer telemetry transits unencrypted.
  • Bearer tokens are HMAC-SHA-256 hashed under a server-side pepper at rest in D1.
  • Admin endpoints use constant-time service-token comparison behind a separate trust boundary from customer-facing endpoints.
  • Telemetry payloads carry no inputs, no labels, no metadata. Count-only verdict events: switch_name, phase, three correctness booleans, timestamp. Full wire spec at docs/legal/telemetry-shape.md.
  • Three opt-outs honored: DENDRA_NO_TELEMETRY=1, per-switch telemetry=False config, /dashboard/settings toggle.
  • Stripe webhook signatures verified before any state mutation, with event-id idempotency. The signing secret comes from env.STRIPE_WEBHOOK_SECRET and is never inline.
  • Compelled-disclosure notification within 24 hours of a lawful request unless legally prohibited. Breach notification within 72 hours from discovery, aligned with GDPR Article 33.

What we don't yet claim

Stated plainly so a procurement reviewer doesn't have to ask.

  • SOC 2: not held today. Type 1 target Q4 2026 (roadmap commitment, not certified state).
  • ISO 27001: not held.
  • HIPAA BAA: template available on request to licensing@b-treeventures.com; not signed by default and we are not a HIPAA-certified processor.
  • 24/7 incident response: we are a small team. Best-effort during business hours; security@b-treeventures.com is monitored daily.
  • Fixed-cadence pen-testing: not scheduled today. Third-party engagement when revenue justifies.

Documents

  • SECURITY.md — disclosure policy, supported versions, scope
  • Data Processing Addendum (template) — GDPR Article 28, Module 2 SCCs by reference
  • Sub-processor list — every third party that touches customer telemetry
  • Access + disclosure policy — personnel access, compelled-disclosure SLA, breach SLA, unmade-claims catalogue
  • Telemetry wire spec — exact on-the-wire format with verification recipe

Reporting a vulnerability

Email security@b-treeventures.com. Do not file a public GitHub issue. You can expect acknowledgement within 72 hours and a triage decision within 5 business days. Critical and high-severity reports get coordinated disclosure on the timeline in SECURITY.md. Anonymous reports welcome.
